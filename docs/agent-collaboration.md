# 多 Agent 协作（Agent Bus & 任务引擎）

本文档说明 Agent Studio 中两项让多个 CLI Agent 协同工作的能力：

1. **Agent Bus（消息转发）** — 把一条消息从一个 Agent 转发给另一个（或多个）Agent。
2. **任务引擎（工作流）** — 把多个 Agent 串成一条线性工作流，前一步的产出自动流入下一步。

两者都建立在同一个底层原语之上：主进程可以向任意 Agent 的真实终端（PTY）写入输入。

> 适用版本：Phase 1。仅支持**线性**协作，运行态不持久化，工作流定义会持久化。
>
> ⚠️ 注意：本文档的「任务引擎」是**用户手工预先编排**的线性流程。若你想要的是「给一句目标、由主代理自动拆解角色并动态创建/调度多个子代理」，那是另一种更进阶的形态，见设计草案 [主代理编排多子代理](agent-orchestration.md)。

## 整体设计

每个 Agent 在主进程里对应一个真实终端（PTY）。Agent 之间没有结构化的消息通道，协作的本质是：

```text
源 Agent 终端输出  ──►  主进程捕获/路由  ──►  写入目标 Agent 终端输入
```

- **Agent Bus** 负责「一次性」的消息路由。
- **任务引擎** 在 Agent Bus 的路由原语之上，增加了「按步编排 + 输出捕获 + 完成检测 + 结果接力」。

```text
                    ┌─────────────────┐
   Claude 终端 ───► │                 │ ───► Codex 终端
   Codex  终端 ───► │   主进程路由中枢  │ ───► Gemini 终端
   Gemini 终端 ───► │ (ptyManager.write) │ ───► …
                    └─────────────────┘
```

## Agent Bus（消息转发）

把当前 Agent 的一段内容发送给同项目内的其它 Agent。

### 如何使用

1. 在任意终端面板，鼠标移到右上角工具栏，点击**转发按钮（→）**。
2. 在弹出框中：
   - **手动输入**消息；或
   - **留空**，则自动使用当前终端中**选中的文本**（便于把某个 Agent 的回答转发出去）。
3. 选择目标：
   - 点击列表中的**某一个 Agent** 单独发送；或
   - 点击「**广播到全部**」发给同项目内其它所有 Agent。

### 投递规则

- 消息以 `[来自 {源 Agent 名}] {内容}` 的形式写入目标终端，并**自动回车提交**，目标 Agent 立即开始处理。
- 目标 CLI **未在运行**时会被跳过，并提示「没有目标接收」。
- 发送成功后提示送达的 Agent 数量。

### 适用场景

- **结果共享**：把 Claude 的分析结论选中后转发给 Codex 去实现。
- **手动协作**：临时给多个 Agent 群发同一条指令。

## 任务引擎（工作流）

把多个步骤串成可保存复用的流程，每步指派一个 Agent + 一段指令，**上一步的终端产出会自动拼接进下一步指令**。

对应路线图示例：

```text
需求分析 → Claude
生成代码 → Codex
代码审查 → Gemini
修复问题 → Reasonix
```

### 如何使用

1. 在左侧活动栏点击**工作流图标**（资源管理器 / 源代码管理 下方的第三个图标）。
2. 点击「**+ 新建**」，填写工作流名称。
3. 逐步添加步骤，每步：
   - 选择**指派的 Agent**（来自当前项目已创建的 Agent）；
   - 填写该步的**指令**；
   - 可上移 / 下移 / 删除步骤。
4. 点击「**保存**」（工作流会持久化，下次可直接复用）。
5. 设置「**空闲秒数**」（判定一步完成的静默阈值，默认 8 秒），点击「**运行**」。

### 运行时

- 引擎把第 1 步指令发给其指派的 Agent 并开始捕获该终端输出。
- 该步被判定完成后（见[完成检测机制](#完成检测机制)），捕获的产出会以如下形式拼进下一步：

  ```text
  [上一步产出]
  …上一个 Agent 的终端输出（已去除 ANSI 颜色码）…

  [任务]
  …本步的指令…
  ```

- 面板上每步有状态圆点：待执行 / 运行中 / 已完成 / 出错。
- 运行中可随时：
  - **立即推进** — 不等空闲，立刻把当前步当作完成并进入下一步；
  - **重试本步** — 重新发送当前步指令；
  - **暂停** — 停止整个运行。
- 若某步指派的 Agent 未在运行，该步标记出错并提示「指派的智能体未在运行」。

### 前置条件

- 工作流里每步指派的 Agent 必须已在当前项目中创建，并且其 CLI 终端处于运行状态。

## 完成检测机制

CLI Agent 没有结构化的「任务完成」信号——它响应完后只是停在提示符等待输入。因此本版采用 **空闲自动 + 手动控制** 的混合策略：

- 引擎每 ~500ms 轮询目标终端的「最近输出时间」。
- 当本步**已经产生过输出**，且输出**静默时长 ≥ 空闲秒数**时，判定该步完成。
- 发送指令时终端的回显会立刻让「已产生输出」成立，Agent 响应结束后的静默触发推进。

> 这是启发式判断，并非 100% 精确：
> - 长耗时步骤（如长时间联网/工具调用中途停顿）可能被提前判定完成 → 调大「空闲秒数」，或用「立即推进 / 重试」兜底。
> - 输出非常零碎的 Agent 可能延迟推进 → 同样可手动「立即推进」。

## 常见问题与限制

- **仅线性流程**：暂不支持分支、并行、条件编排。
- **运行态不持久化**：应用重启后，正在进行的运行会中断；但**工作流定义会持久化**，可重新运行。
- **不跨项目**：消息转发与工作流都限定在同一项目内的 Agent 之间。
- **自动重试**：当前仅提供手动「重试本步」；全自动重试在后续规划中。
- **输出捕获上限**：单步捕获约 100KB，超出时丢弃最旧内容（转发给下一步前会去除 ANSI 颜色码）。

更多后续规划见根目录 [README.md](../README.md) 的「后续规划」一节。
