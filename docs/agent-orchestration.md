# 主代理编排多子代理（Master-Orchestrated Multi-Agent）设计草案

> 状态：**设计草案（尚未实现）**。本文档描述目标形态；当前已实现的是较简单的[线性工作流（任务引擎）](agent-collaboration.md)，两者的区别见[文末对比](#与现有线性任务引擎的区别)。

## 背景与目标

现有的[任务引擎](agent-collaboration.md)需要用户**手工预先编排**每个步骤（指派哪个 Agent、写什么指令、按固定顺序串行执行）。

本设计要实现的是一种**自顶向下、由主代理驱动**的协作模式：

- 用户只给出**一句高层目标**，例如：
  > 「给我写一个弹幕统计可视化的项目」
- 由一个**主代理（Orchestrator）自动**完成：
  1. 把目标**拆解**成一个软件从需求到落地所需的**各种角色**；
  2. 为每个角色**动态创建一个子代理**，分配各自的任务；
  3. 标注子代理之间的**依赖关系**（谁必须等谁完成后才能开始）；
  4. **并发**运行相互独立的子代理，按依赖**调度**其余子代理；
  5. 持续**监督**：收集子代理产出、在它们之间接力结果，必要时新增 / 重试 / 调整子代理。

一句话：**主代理是"项目负责人"，子代理是被它临时组建并指挥的"团队成员"。**

---

## 核心概念

| 概念 | 说明 |
| --- | --- |
| **主代理 / Orchestrator** | 接收高层目标，产出"编排计划"，并在运行期指挥子代理。可以是一个被专门提示的 CLI Agent，也可以由 App 内的编排引擎承担一部分确定性逻辑（见[与现有架构的衔接](#与现有架构的衔接)）。 |
| **子代理 / Sub-Agent** | 承担单一角色的真实 CLI Agent（Claude / Codex / Gemini 等），运行在各自的终端（PTY）中。 |
| **角色 / Role** | 子代理在项目中的职责，如 需求分析、架构、前端、后端、测试。 |
| **任务图 / Task DAG** | 子代理任务之间的有向无环依赖图。无依赖的节点并发执行，有依赖的节点等待前置完成。 |
| **编排计划 / Plan** | 主代理产出的结构化方案：包含子代理列表（角色、类型、任务指令）与它们之间的依赖。 |

## 端到端工作流程

```text
用户目标
  │  「给我写一个弹幕统计可视化的项目」
  ▼
┌────────────────────────────────────────────┐
│ 主代理 Orchestrator                          │
│  1. 角色拆解 + 任务分配 + 依赖标注             │
│  2. 产出「编排计划」(结构化)                    │
└────────────────────────────────────────────┘
  │  Plan { 子代理[], 依赖[] }
  ▼
┌────────────────────────────────────────────┐
│ 编排引擎 / 调度器                              │
│  3. 据计划创建子代理终端                       │
│  4. 按 DAG 调度：就绪即并发启动，未就绪则等待    │
│  5. 捕获产出 → 接力给下游 / 回传主代理          │
└────────────────────────────────────────────┘
  │  各子代理产出 + 状态
  ▼
┌────────────────────────────────────────────┐
│ 主代理监督（可选的运行期动态控制）              │
│  6. 审阅产出，决定新增/重试/调整后续子代理       │
└────────────────────────────────────────────┘
```

## 示例：「写一个弹幕统计可视化的项目」

主代理可能拆解出如下角色与依赖：

| 子代理（角色） | 指派 CLI | 任务 | 依赖（等待） |
| --- | --- | --- | --- |
| ① 需求分析 PM | Claude | 梳理弹幕统计可视化的功能清单、数据指标、页面 | — |
| ② 架构设计 | Claude | 定技术栈、模块划分、数据结构、接口约定 | ① |
| ③ 后端开发 | Codex | 实现弹幕采集 / 解析 / 统计 API | ② |
| ④ 前端开发 | Codex | 实现页面骨架与交互 | ② |
| ⑤ 数据可视化 | Codex | 实现图表（趋势、热词、Top 用户等） | ④ |
| ⑥ 集成联调 | Gemini | 前后端对接、跑通端到端 | ③, ⑤ |
| ⑦ 测试 QA | Gemini | 用例、边界、回归 | ⑥ |
| ⑧ 文档 | Claude | README、使用与部署说明 | ⑥ |

对应的并发调度（DAG）：

```text
        ┌──► ③ 后端 ──────────────┐
① ──► ② │                         ├──► ⑥ 集成 ──► ⑦ 测试
        └──► ④ 前端 ──► ⑤ 可视化 ──┘            └──► ⑧ 文档
```

- ① 完成后 ② 启动；
- ② 完成后 **③ 与 ④ 并发**启动；
- ④ 完成后 ⑤ 启动；
- ③ 与 ⑤ **都**完成后 ⑥ 启动；
- ⑥ 完成后 **⑦ 与 ⑧ 并发**启动。

## 编排计划（Plan）数据模型（建议）

主代理需要产出一份机器可读的计划，建议结构如下（字段名待定）：

```jsonc
{
  "goal": "写一个弹幕统计可视化的项目",
  "subAgents": [
    {
      "key": "pm",                 // 计划内唯一标识，用于依赖引用
      "role": "需求分析",
      "type": "claude",            // 指派的 CLI 类型
      "task": "梳理弹幕统计可视化的功能清单、数据指标与页面结构…",
      "dependsOn": []              // 依赖的其它 subAgent.key，空=可立即开始
    },
    { "key": "arch", "role": "架构设计", "type": "claude", "task": "…", "dependsOn": ["pm"] },
    { "key": "be",   "role": "后端开发", "type": "codex",  "task": "…", "dependsOn": ["arch"] },
    { "key": "fe",   "role": "前端开发", "type": "codex",  "task": "…", "dependsOn": ["arch"] },
    { "key": "viz",  "role": "数据可视化", "type": "codex", "task": "…", "dependsOn": ["fe"] },
    { "key": "int",  "role": "集成联调", "type": "gemini", "task": "…", "dependsOn": ["be", "viz"] },
    { "key": "qa",   "role": "测试",     "type": "gemini", "task": "…", "dependsOn": ["int"] },
    { "key": "doc",  "role": "文档",     "type": "claude", "task": "…", "dependsOn": ["int"] }
  ]
}
```

- 每个子代理在运行时映射为一个真实终端（PTY）。
- `dependsOn` 形成 DAG；引擎据此调度。
- 上游产出在下游启动时拼接进其任务指令（沿用现有[结果接力](agent-collaboration.md#任务引擎工作流)机制）。

## 控制模式

用户描述的两种控制方式都应支持：

1. **静态依赖声明（创建时编排）**
   主代理在一开始就把整张 DAG（谁等谁）标注清楚，引擎据此自动调度，主代理之后较少干预。**适合需求明确的项目，确定性强。**

2. **全动态控制（运行期编排）**
   主代理在运行期根据子代理的产出，**逐步决定**下一步创建哪个子代理、让谁做什么、是否重试或改派。计划不是一次成型，而是边跑边长。**适合探索性任务，灵活但更依赖主代理判断。**

> 可以先实现「静态依赖声明」（更确定、易验证），再在其上叠加「动态控制」（主代理可在运行期追加 / 修改节点）。

## 调度与并发

- **拓扑调度**：维护每个子代理的状态（待依赖 / 就绪 / 运行中 / 完成 / 失败）。当一个节点的全部 `dependsOn` 都为"完成"时变为"就绪"并启动。
- **并发上限**：同时运行的子代理数量设上限（受机器资源、CLI 配额限制），超出的就绪节点排队。
- **完成检测**：沿用任务引擎的[空闲自动 + 手动控制](agent-collaboration.md#完成检测机制)，并保留主代理/用户的「立即推进 / 重试 / 终止」干预。
- **失败处理**：某节点失败时，可重试该节点、或由主代理改派/降级，下游保持等待直至其完成或被跳过。
- **共享工作区**：所有子代理在**同一个项目目录**下工作（通过文件系统协作），降低对"把全部上下文塞进提示词"的依赖。

## 与现有架构的衔接

可大量复用 Agent Studio 现有基建：

| 需要 | 复用 / 新增 |
| --- | --- |
| 子代理 = 真实终端 | **复用** `src/main/ptyManager.ts`（每个子代理一个 PTY）、`Agent` 实体 |
| 向子代理下达指令 | **复用** `ptyManager.write()`（Agent Bus / 任务引擎同款路由原语） |
| 捕获子代理产出 | **复用** `ptyManager` 的输出捕获 + ANSI 清洗 |
| 完成检测 | **复用** `src/main/taskEngine.ts` 的空闲轮询逻辑 |
| 主代理 | **新增**：一个被专门提示、要求输出上述 Plan JSON 的 CLI Agent；App 解析其输出 |
| 计划解析 | **新增**：从主代理终端输出中提取并校验 Plan JSON |
| DAG 调度器 | **新增**：拓扑调度、并发控制、依赖门控（在 `taskEngine` 之上扩展或新建 `orchestrator.ts`） |
| 可视化 | **新增**：DAG / 角色看板 UI（子代理状态、依赖连线、产出预览） |

一种现实的分工：**主代理（CLI Agent）负责"想"出计划（角色拆解、依赖）**，**App 内的确定性调度器负责"执行"计划**（创建终端、按 DAG 调度、接力产出），二者结合，既利用 LLM 的规划能力，又保证调度的可控与可观测。

## 与现有线性任务引擎的区别

| 维度 | 现有线性任务引擎（已实现） | 主代理编排（本草案） |
| --- | --- | --- |
| 谁来编排 | **用户**手工预先定义每步 | **主代理**根据一句目标自动拆解 |
| 结构 | 线性串行（步骤 1→2→3） | **DAG**，支持并发与多重依赖 |
| Agent 来源 | 用户预先创建好的固定 Agent | 主代理**动态创建**子代理（按角色） |
| 并发 | 否（一次一步） | **是**（无依赖的子代理并行） |
| 运行期调整 | 否 | 可选的**动态控制**（主代理边跑边编排） |
| 适用 | 简单、确定的固定流程 | 复杂、需多角色协作的项目 |

> 线性任务引擎仍然有用（适合简单固定流程），主代理编排是其超集方向，二者可共存。

## 待确认的设计决策

实现前需要敲定：

1. **主代理形态**：用某个 CLI Agent 充当主代理（提示它输出 Plan JSON）？还是 App 内置一个调用 Claude API 的编排器？
2. **计划如何落地**：从终端输出里解析 JSON（需稳定的输出约定）？还是给主代理一个专用的"提交计划"通道？
3. **静态 vs 动态**：第一版先做哪种（建议静态 DAG 优先）？
4. **并发上限**与资源/配额策略。
5. **子代理协作介质**：纯文件系统共享？是否需要更结构化的产出交接（如约定每个子代理把成果写到指定文件）？
6. **人工介入点**：哪些环节需要用户确认（如计划生成后先给用户审阅再执行）？
7. **失败 / 死循环兜底**：重试次数、超时、主代理跑偏时的熔断。

---

## 建议的实现阶段

1. **阶段 A**：主代理产出**静态 Plan（DAG）** → App 解析 → 调度器据 DAG 创建子代理终端并并发执行、按依赖门控、结果接力。
2. **阶段 B**：DAG 可视化看板（角色、状态、依赖、产出预览）+ 人工审阅/干预。
3. **阶段 C**：**动态控制**——主代理在运行期追加 / 修改 / 重试子代理节点。
4. **阶段 D**：失败恢复、并发与配额策略、运行历史持久化（配合后续 SQLite）。
